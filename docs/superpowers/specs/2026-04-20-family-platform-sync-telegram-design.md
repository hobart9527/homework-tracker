# 家庭多平台同步与 Telegram 通知设计

**日期：** 2026-04-20
**范围：** 基于当前 `homework-tracker` 项目，为单家庭场景增加多孩子、多平台账号自动同步、激进型自动打卡、Telegram 家长通知，以及录音类作业推送到个人微信群的 Beta 通道。

## 目标

这一阶段聚焦三个结果：

- 家长每天能收到一条基于真实作业和平台学习记录生成的 Telegram 完成情况通知
- 系统能自动抓取主流国际学校学习平台的学习记录，并自动完成对应作业打卡
- 录音类作业在孩子提交后，能够额外进入个人微信群推送链路，方便老师查看

本次设计服务的目标用户结构是：

- 一个家庭
- 多个孩子
- 每个孩子可绑定多个平台账号
- 一个家长 Telegram 接收人

## 设计范围

### 本期包含

- Telegram 日报、周报和基础即时通知
- 每个孩子绑定多个学习平台账号
- 平台学习记录定时同步
- 激进型自动打卡
- 学习内容留痕和来源展示
- 录音类作业进入个人微信群推送任务队列

### 本期不包含

- AI 辅导建议
- 多家长接收人
- 家长端自定义通知编排
- 所有作业类型都自动发个人微信群
- 把微信群推送失败纳入主作业完成链路

## 核心原则

- 通知先有用：先把完成情况准时发给家长，再考虑建议生成
- 学习记录先统一：无论来自哪个平台，先转成统一学习事件，再驱动作业打卡
- 激进打卡但保留依据：尽量少让家长确认，但必须能看见“为什么被判定为已完成”
- 高风险能力隔离：个人微信群自动推送是独立 Beta 通道，不能拖垮主业务链路
- 平台差异向内收敛：平台层各自复杂，进入系统后统一数据口径

## 外部能力边界

本次设计对外部平台做如下假设：

- Telegram 推送采用 Bot API，能力稳定，可作为正式通知通道
- IXL、Raz-Kids、Epic、Khan Academy 的学习记录同步不假设存在统一公开 API
- 平台同步器允许采用“平台专属采集方式”，但进入系统后必须产出统一学习事件
- 个人微信群自动推送录音采用独立桥接服务处理，不与 Next.js 主服务混跑

## 信息架构

这一阶段新增四个子系统：

1. 家长通知中心
2. 多平台账号绑定与同步调度中心
3. 自动打卡与学习内容留痕中心
4. 微信群录音推送 Beta 通道

其中 1、2、3 共同构成主线闭环，4 为独立附加链路。

## 子系统设计

### 1. 家长通知中心

通知中心服务单一 Telegram 接收人，但聚合同一家庭下所有孩子的数据。

这里需要额外明确对象层级：首版 Telegram 是`家庭级家长通知通道`，主要负责家庭日报、自动完成即时通知和同步失败通知。它不是单条作业的消息路由目标，也不应该和孩子级、作业级的投递规则混成同一个配置模型。

首版通知类型：

- `家庭日报`
  - 每个孩子当天已完成作业
  - 哪些完成来自外部平台自动同步
  - 哪些作业仍未完成
  - 当天录音类作业是否已提交
- `家庭周报`
  - 每个孩子一周完成率
  - 自动打卡次数
  - 平台学习活跃度摘要
- `即时通知`
  - 自动打卡成功
  - 当天仍有未完成作业
  - 平台同步失败

通知只发送事实型内容，不生成辅导建议。文案以短摘要为主，避免打扰。

首版 Telegram 配置建议拆成两类：

- `数据库配置`
  - `telegram_chat_id`
  - `telegram_recipient_label`
- `运行时配置`
  - `TELEGRAM_BOT_TOKEN`

原因是 `chat_id` 和接收人备注属于家庭级业务配置，需要按家庭保存；`bot token` 属于通道密钥，更适合作为运行时环境配置而不是普通家长资料字段。

### 2. 多平台账号绑定与同步调度中心

每个孩子可以绑定多个平台账号，例如：

- IXL
- Raz-Kids
- Epic
- Khan Academy

绑定完成后，系统按固定计划自动同步，不要求家长每天手动触发。每个平台拥有独立同步器，但必须通过统一调度器执行，统一写入同步结果和学习事件。

调度策略：

- 支持周期性计划任务
- 支持家长手动补触发“立即同步”
- 同一孩子、同一平台在同一时段内只允许一个活动同步任务
- 同步失败保留错误摘要，进入家长通知中心

### 3. 自动打卡与学习内容留痕中心

自动打卡采用用户确认过的激进型策略：

- 只要平台当天存在明确学习记录，就优先尝试完成对应作业
- 不把“待家长确认”作为默认路径
- 每次自动打卡都必须保留具体学习内容

自动打卡规则分三类：

- `直接绑定`
  - 本地作业明确绑定平台任务或技能 ID
- `平台 + 类型匹配`
  - 例如 Epic 阅读记录自动完成“每日英文阅读”
- `阈值匹配`
  - 例如达到某个时长、次数、测验完成条件后自动打卡

自动打卡结果分为：

- `auto_completed`
- `partially_completed`
- `unmatched`

其中 `auto_completed` 和 `partially_completed` 都必须在记录中附带：

- 平台名
- 同步时间
- 学习标题
- 任务或技能标识
- 时长、成绩或完成状态
- 原始来源引用

首版基础阈值已确认：

- 如果同步回来的有效学习时长大于等于本地作业要求时长，则直接判定为 `auto_completed`
- 不再默认要求家长二次确认
- 当多条学习记录同时命中同一条作业时，采用“最早命中优先”作为主证据选择规则

### 4. 微信群录音推送 Beta 通道

录音类作业提交成功后，主系统先完成以下动作：

1. 正常完成作业内的打卡提交
2. 保存录音附件
3. 将该录音纳入 Telegram 通知摘要
4. 生成一条微信群推送任务

随后由独立的微信群桥接服务处理该任务。

桥接服务职责：

- 拉取待发送录音任务
- 向指定个人微信群发送文本说明和录音文件
- 回写发送状态、失败原因和重试次数

关键约束：

- 微信群发送失败不能回滚作业完成状态
- 只处理录音类作业
- 不承担图片、文本、视频等其他任务类型的自动群推送
- 孩子级或作业级消息路由主要为微信群 bridge 这类投递目标服务，不作为家庭 Telegram 摘要通道的配置来源

## 数据流

### 主线数据流

1. 家长为某个孩子绑定平台账号
2. 调度器按计划触发平台同步器
3. 平台同步器抓取学习记录
4. 学习记录统一落成 `learning_events`
5. 自动打卡引擎扫描当日作业与学习事件
6. 符合条件的作业自动完成或部分完成
7. 作业完成详情保留具体学习内容
8. 通知中心聚合家庭当天数据并推送 Telegram

### 录音 Beta 数据流

1. 孩子提交录音类作业
2. 主系统写入作业完成记录和录音附件
3. 主系统生成 `voice_push_tasks`
4. 微信桥接服务拉取任务并尝试发群
5. 发送状态回写任务表

## 产品行为细节

### Telegram 日报

日报按家庭维度发送一条，结构建议为：

- 标题：今天的家庭作业完成情况
- 每个孩子一个小节
- 已完成作业列表
- 自动打卡来源摘要
- 未完成作业列表
- 录音作业提交状态

发送时间固定在每天晚间 `21:30-22:00` 窗口内。

首版日报接收目标来自家庭级 Telegram 配置，而不是某个孩子或某条作业的消息路由规则。

### Telegram 周报

周报按家庭维度发送一条，聚焦：

- 每个孩子本周完成率
- 平台学习活跃天数
- 自动打卡次数
- 未完成任务累计数

### 平台同步可见性

家长端需要能查看：

- 每个孩子绑定了哪些平台账号
- 最近一次同步成功或失败时间
- 最近一次同步抓到了哪些学习内容

首批落地平台优先级为：

1. IXL
2. Khan Academy
3. Raz-Kids
4. Epic

### 自动打卡可解释性

家长端需要能查看某条作业的自动完成依据，例如：

- 平台：Epic
- 内容：Read to Me - Animals at Night
- 时长：18 分钟
- 同步时间：19:42

避免出现“状态变了，但家长不知道依据”的体验问题。

## 错误处理

### 平台同步失败

- 不中断其他孩子或其他平台的同步
- 写入同步任务失败摘要
- 在 Telegram 即时通知中提示“某平台同步失败”

### 同步频率

首版不同步追求接近实时，而采用“每天固定几次批量同步”的策略。每个平台账号按照预设批次运行，同一天内可以多次同步，但不按分钟级事件驱动。

已确认的首个主同步窗口是放学后，家庭实际可用时间一般从下午 `15:30` 之后开始。实施计划应优先围绕放学后批次设计。

### 自动打卡失败

- 不回滚学习事件
- 保留未匹配状态
- 在家长端显示“已同步但未自动打卡”

### 微信群录音发送失败

- 不影响作业打卡完成
- 写入 `voice_push_tasks.failure_reason`
- 支持有限重试
- 超过重试阈值后停止自动重试

## 边界条件

### 学习记录去重

系统必须把重复同步到的同一条外部学习记录视为一个逻辑事件。如果同一孩子、同一平台、同一来源引用被重复同步，只允许落成一条规范化 `learning_event`。重复导入不得触发重复自动打卡、重复加分或重复 Telegram 即时通知。

### 多条学习记录命中同一作业

同一天可能有多条学习记录同时命中同一条作业。在这种情况下，只允许一个主命中结果驱动作业状态变化。其他命中记录应作为补充证据保留，但不得重复完成同一条作业，也不得重复触发完成副作用。

首版主证据选择规则已确认：

- 采用“最早命中优先”
- 最早满足匹配条件的 `learning_event` 作为主证据
- 后续命中记录仅作为补充证据保留，不再重复改变作业状态

### 手动状态优先级

手动作业状态仍然具有最高优先级。如果某条作业已经被手动完成，后续同步到的学习记录可以补充审计证据，但不能覆盖已存在的完成状态，也不能再生成第二次完成事件。如果某条作业已被手动跳过、豁免或通过其他人工路径处理，自动打卡引擎不得静默反转该结果。

### 部分证据与完整完成的区别

平台学习证据并不总能满足整条作业要求。当作业还要求录音、图片或其他附件证明时，单纯的学习时长不得自动满足全部要求。这类情况下，系统应保留学习证据，并将作业标记为 `partially_completed`，而不是 `auto_completed`。

### 跨天归属

所有导入学习记录都必须以家庭配置的本地时区作为归属依据，而不是直接使用平台原始时间。接近午夜的学习记录在匹配前必须完成时区归一化，确保晚间学习被归到正确的作业日期。

### 回补同步行为

如果平台在白天同步失败、之后恢复，系统可以回补当天学习记录，并仍然对正确日期的作业执行自动打卡。回补行为必须更新审计轨迹，但不能制造重复完成副作用。

### 异常时长值

用于阈值规则的学习时长必须先通过有效性校验。缺失、零值、负值或明显异常的时长，不能直接触发 `auto_completed`。这类记录可以保留用于审计，但在被归一化或人工确认前，不应作为有效完成证据。

### 单孩子同平台多账号并存

首版允许同一孩子在同一平台下并存多个账号，例如家庭账号和学校账号。系统必须保留账号维度，以便区分同步来源、审计轨迹和自动打卡证据。多账号并存不得导致学习记录互相覆盖，也不得因为来源不同而重复触发同一条作业的完成副作用。

### 录音桥接幂等

同一条 `voice_push_task` 在桥接不稳定时允许重试，但重试不得导致同一份音频被重复发送到目标群。即使桥接回执延迟或缺失，录音推送链路也必须保持幂等。

## 兼容性风险

### 平台数据结构差异

IXL 和 Khan Academy 不应被视为完全同构的数据源。某个平台可能更擅长暴露时长和技能级别证据，另一个平台可能更偏向课程或活动完成。共享规则“学习时长大于等于作业要求时长即完成”可以作为通用底座，但仍然需要平台专属匹配规则。

### 凭据和会话生命周期

首版允许账号密码与会话托管，这会引入真实运行风险：密码变更、会话过期、MFA 提示、验证码挑战和异常登录保护。系统因此必须支持类似 `attention_required` 的中间状态，而不能只用 `success` 和 `failed` 两种结果覆盖全部情况。

### Telegram 送达差异

Telegram 投递必须容忍接收端和平台端差异，包括无效 chat ID、限流、静音、屏蔽 Bot 或权限被撤销。即时通知与夜间汇总必须被视为两类独立投递，拥有各自的重试和失败状态。

### 家庭电脑与群晖 NAS 的运行环境差异

微信桥接运行环境并不等价。家里常开电脑更适合作为桌面自动化或 GUI 驱动的最终发送端，群晖 NAS 更适合作为队列处理、调度和桥接协调宿主。首版不应假定这两类环境都能承担相同的最终发送职责。

### Telegram 即时通知与日报的内容重叠

同一条作业完成记录可能同时出现在即时通知和晚间家庭日报中。这种重叠只有在被视为产品有意行为时才是可接受的，而不能被误认为重复通知 bug。实施计划需要显式决定日报是否汇总所有当天完成记录，还是排除已即时发出的项目。

### 平台自身不稳定性

由于本次设计不假设所有目标平台都有稳定公开 API，同步行为可能会受到页面结构变化、导出格式变化或反自动化策略影响。连接器失败必须优雅降级，不得破坏作业状态，也不得阻断其他平台的同步。

## 第一期上线门槛

第一期不是只看功能是否存在，而是看用户流程、正确性护栏和验证标准是否同时到位。任何能力如果只有可见功能、没有护栏和可验证标准，都不应被视为第一期完成。

本次第一期必须同时包含：

- 首批支持平台 IXL 和 Khan Academy 的定时同步
- 基于同日学习证据的激进型自动打卡
- Telegram 即时完成通知与夜间家庭日报
- 录音类作业桥接任务创建与发送处理
- 学习记录去重
- 重复同步导致的重复自动完成防护
- 手动状态优先规则
- 证据不足时的部分完成处理
- 凭据失效和会话过期的可见失败状态
- Telegram 的重试与去重机制
- 录音桥接的重试与幂等机制
- 自动打卡依据的审计可见性
- rollout 前可执行的验收检查
- 家庭级 Telegram 通道与孩子/作业级消息路由的明确分层
- Telegram Bot Token 作为运行时密钥而不是普通家长资料字段

如果以下任一情况仍然成立，则第一期不应视为可上线：

- 同一平台记录会生成重复学习事件
- 同一条作业会因为重复同步被重复自动完成、重复加分或重复通知
- 手动完成结果会被后续自动规则静默覆盖
- 需要附件证明的作业会仅凭学习时长被判定为完整完成
- 平台凭据或会话失效会伪装成“同步成功但没有数据”

在范围上，第一期仍然可以保持克制：一个家庭、多个孩子、一个 Telegram 接收人、首批 IXL/Khan、以及仅针对录音作业的 Beta 微信桥接。但在这个范围内，上线门槛必须是“正确性优先，再扩能力”。

## 验收标准

### 平台同步验收

系统必须允许一个家庭下的多个孩子分别绑定 IXL 和 Khan Academy 账号，且同一孩子同一平台允许同时存在多个账号。在预设批次时间到达后，系统必须能够为符合条件的 child-platform-account 绑定创建并执行 sync job。系统必须阻止同一孩子、同一平台、同一账号在同一执行窗口内并发生成重复 sync job。重复导入同一账号来源下的同一平台记录不得创建重复 `learning_events`。当凭据或会话失效时，平台账号必须进入可见的失败或 attention 状态，而不能表现成“同步成功但零数据”。

### 自动打卡验收

当导入的学习证据包含有效学习时长，且时长大于等于作业要求时长，并且没有额外附件要求阻止完成时，作业必须进入 `auto_completed`。当学习证据能证明有进展但不足以满足全部要求时，作业必须进入 `partially_completed`。重复导入相同证据不得造成重复完成、重复加分或重复状态变化。当多条学习记录同时命中同一条作业时，系统必须以最早命中的记录作为主证据，后续命中只能补充证据，不能再次改变状态。家长必须可以查看每次自动完成的依据，包括平台、账号来源、学习标题、时长或完成值，以及同步时间。手动完成结果必须保持稳定，不能被后续自动规则覆盖。

### Telegram 验收

系统必须在作业被自动完成时发送一条即时 Telegram 通知，并在每晚 `21:30-22:00` 窗口内发送一条家庭日报。任一消息类型的投递失败都不得影响作业完成状态。重试不得让同一逻辑完成事件生成重复即时通知。家庭日报必须能够正确聚合同一家庭下多个孩子的当天状态摘要。
首版 Telegram 接收目标必须来自家庭级配置，其中 `telegram_chat_id` 和接收人备注保存在数据库里，而 `Telegram Bot Token` 来自运行时配置。家庭 Telegram 摘要通道不得依赖 `message_routing_rules` 决定投递目标。

### 录音桥接验收

当录音类作业提交成功时，系统必须创建一条引用最终音频附件的 `voice_push_task`。桥接发送失败不得改变原始作业提交成功结果。重试失败任务时，不得向目标群重复发送同一条音频。任务状态至少必须能查询到 `pending`、`retrying`、`sent`、`failed` 这几类结果。

### 运行可观测性验收

运维或维护者必须能够查看近期平台同步状态、同步失败、去重行为，以及待发送或失败的录音桥接任务。系统必须保留足够的审计细节，用于解释一条作业为什么被自动完成、部分完成或未匹配。

## 安全与边界

- 平台账号绑定信息必须与孩子维度隔离
- 不允许一个同步任务串写到另一个孩子的学习记录
- 通知只发给当前家庭唯一 Telegram 接收人
- 微信群桥接服务的失败不应暴露到孩子端主流程
- 原始平台学习记录应保留引用和原始摘要，便于后续审计

平台账号首版允许采用账号密码和会话托管方式，但实现时必须将凭据和会话信息与普通学习记录分离存储。

## 实施拆分建议

这一阶段虽然属于同一个产品主题，但实现上建议拆成两个计划：

### Plan A：主线能力

- Telegram 家长通知中心
- 多平台账号绑定
- 定时同步调度
- 学习事件统一入库
- 激进型自动打卡
- 学习内容留痕

### Plan B：Beta 能力

- 录音类作业微信群推送桥接
- 推送任务状态管理
- 重试和失败回写

这样可以先确保主线价值上线，再处理高风险通道。

## 验证策略

### 单元验证

- 平台账号绑定模型与状态流转
- 自动打卡规则匹配
- Telegram 日报/周报载荷生成
- 微信群录音任务状态机

### 集成验证

- 一个家庭下两个孩子绑定不同平台账号
- 同一天同步多平台记录后自动完成多条作业
- Telegram 日报正确聚合两个孩子的数据
- 录音作业提交后生成微信群推送任务

### 冒烟验证

- 家长成功绑定平台账号
- 定时同步执行成功
- 自动打卡结果在家长端可见
- Telegram 能收到完成情况消息
- 录音任务在桥接队列中可见

## 已确认产品口径

- 首批平台优先：IXL、Khan Academy
- 同步方式：每天固定几次批量同步
- 首个主同步窗口：放学后 `15:30` 之后
- Telegram 策略：作业完成即时通知 + 晚间 21:30-22:00 家庭日报
- 自动打卡阈值：同步学习时长大于等于作业要求时长即自动完成
- 平台绑定方式：账号密码 + 会话托管
- 同平台多账号：允许并存，例如家庭账号 + 学校账号
- 主证据规则：多条记录命中同一作业时采用最早命中优先
- 微信群录音推送：首版只发送音频文件，不附带固定说明文案
- 微信桥接运行环境：家里常开电脑或群晖 NAS 设备

## 结论

这次设计的核心不是“把更多通道堆进系统”，而是先把“家庭作业与外部学习记录统一起来”。只要统一学习事件、自动打卡和 Telegram 通知先跑通，平台能力和微信群能力都可以在统一框架下逐步扩展，而不会把主系统拖成一组相互耦合的脚本。
